Scenario 03: CI/CD Pipeline Security
The Challenge
You need to securely manage sensitive data, such as database passwords or third-party API keys, within a CI/CD pipeline (like Jenkins or GitLab). Hardcoding these secrets in the source code or printing them in build logs is a massive security risk.

The Architectural Solution
Secret Managers: Store the sensitive data in a centralized, encrypted vault like AWS Secrets Manager or HashiCorp Vault.

IAM Roles (Not Access Keys): Instead of giving the CI/CD pipeline long-lived AWS Access Keys, use an IAM Role. If Jenkins is on an EC2 instance, use an Instance Profile. If using GitHub/GitLab, use OIDC (OpenID Connect) to assume a temporary IAM role.

Secret Masking: Ensure the CI/CD tool is configured to mask or hide any output that matches a known secret.

Phase 1: Infrastructure Setup (Terraform)
Objective
Provision an AWS Secrets Manager secret and create an IAM Role that grants read-only access to that specific secret. We will simulate a scenario where an EC2-based Jenkins server needs to read a database password.

Step 1: Directory Setup
Bash
mkdir Scenario-03-CICD-Security
cd Scenario-03-CICD-Security
touch main.tf
Step 2: Define Infrastructure (Terraform)
Secrets Manager Setup:

Define the secret container.

Define the actual sensitive payload (e.g., a dummy database password).

IAM Configuration:

Create an IAM Policy that allows the secretsmanager:GetSecretValue action, restricted only to the specific secret.

Create an IAM Role for the Jenkins EC2 instance and attach the policy.

Create an IAM Instance Profile to attach the role to the server.

Terraform
# main.tf

provider "aws" {
  region = "us-east-1"
}

# 1. Create the Secret in AWS Secrets Manager
resource "aws_secretsmanager_secret" "db_password" {
  name        = "prod/app/db_password"
  description = "Production database password for the application"
}

# 2. Store the actual secret value (In a real scenario, use variables or tfvars)
resource "aws_secretsmanager_secret_version" "db_password_value" {
  secret_id     = aws_secretsmanager_secret.db_password.id
  secret_string = jsonencode({
    username = "admin"
    password = "SuperSecretPassword123!"
  })
}

# 3. Create an IAM Policy allowing read access to this specific secret
resource "aws_iam_policy" "secret_read_policy" {
  name        = "JenkinsSecretReadPolicy"
  description = "Allows Jenkins to read the prod DB password"

  policy = jsonencode({
    Version = "2012-10-17"
    Statement = [
      {
        Effect   = "Allow"
        Action   = "secretsmanager:GetSecretValue"
        Resource = aws_secretsmanager_secret.db_password.arn
      }
    ]
  })
}

# 4. Create an IAM Role for the Jenkins EC2 Server
resource "aws_iam_role" "jenkins_role" {
  name = "jenkins_ec2_role"

  assume_role_policy = jsonencode({
    Version = "2012-10-17"
    Statement = [
      {
        Action = "sts:AssumeRole"
        Effect = "Allow"
        Principal = {
          Service = "ec2.amazonaws.com"
        }
      }
    ]
  })
}

# 5. Attach the Policy to the Role
resource "aws_iam_role_policy_attachment" "jenkins_secret_attach" {
  role       = aws_iam_role.jenkins_role.name
  policy_arn = aws_iam_policy.secret_read_policy.arn
}

# 6. Create the Instance Profile to attach to the EC2 instance
resource "aws_iam_instance_profile" "jenkins_profile" {
  name = "jenkins_instance_profile"
  role = aws_iam_role.jenkins_role.name
}

Phase 2: CI/CD Pipeline Implementation (Concept)
If you were writing the Jenkinsfile for this scenario, you would securely fetch the secret using the AWS CLI, utilizing the permissions granted by the Instance Profile we just created.

Groovy
pipeline {
    agent any
    stages {
        stage('Fetch Secrets') {
            steps {
                script {
                    // The EC2 instance automatically uses its IAM Role to authenticate
                    def secretJson = sh(script: "aws secretsmanager get-secret-value --secret-id prod/app/db_password --query SecretString --output text", returnStdout: true).trim()

                    // Parse the JSON (Requires Jenkins Pipeline Utility Steps plugin)
                    def secretProps = readJSON text: secretJson
                    env.DB_USER = secretProps.username
                    env.DB_PASS = secretProps.password
                }
            }
        }
        stage('Deploy') {
            steps {
                // Example of using the secret securely
                sh "echo Deploying with DB User: ${env.DB_USER}"
                // Note: The password should never be echoed!
            }
        }
    }
}

Paste the following code directly into the Script box:

Groovy
pipeline {
    agent any
    stages {
        stage('Fetch Secrets') {
            steps {
                script {
                    // Jenkins uses the EC2 Instance Profile to authenticate implicitly
                    def secretJson = sh(script: "aws secretsmanager get-secret-value --secret-id prod/app/db_password --query SecretString --output text", returnStdout: true).trim()

                    // Parse the JSON payload
                    def secretProps = readJSON text: secretJson

                    // Set as environment variables for the current build
                    env.DB_USER = secretProps.username
                    env.DB_PASS = secretProps.password

                    echo "Successfully fetched secrets from AWS!"
                }
            }
        }
        stage('Deploy Application') {
            steps {
                // We echo the username to verify it worked. 
                // NEVER echo the password in a real production pipeline.
                sh "echo Connecting to database with User: ${env.DB_USER}"
            }
        }
    }
}
Note: This script requires the Pipeline Utility Steps plugin in Jenkins to use the readJSON function. If it fails on that line, you can install the plugin via Manage Jenkins -> Manage Plugins.

We will add an EC2 instance to your existing main.tf file. This code will automatically fetch the latest Amazon Linux AMI, create a Security Group for ports 22 and 8080, attach the IAM role we just created, and use a user_data script to install and start Jenkins on boot.

Step 1: Append this code to your main.tf
Terraform
# --- ADD THIS TO YOUR EXISTING main.tf ---

# 7. Data source to get the latest Amazon Linux 2023 AMI
data "aws_ami" "amazon_linux_2023" {
  most_recent = true
  owners      = ["amazon"]
  filter {
    name   = "name"
    values = ["al2023-ami-2023.*-x86_64"]
  }
}

# 8. Security Group for Jenkins
resource "aws_security_group" "jenkins_sg" {
  name        = "jenkins_sg"
  description = "Allow port 8080 for Jenkins UI and 22 for SSH"

  ingress {
    from_port   = 22
    to_port     = 22
    protocol    = "tcp"
    cidr_blocks = ["0.0.0.0/0"] # In a real setup, restrict this to your IP
  }

  ingress {
    from_port   = 8080
    to_port     = 8080
    protocol    = "tcp"
    cidr_blocks = ["0.0.0.0/0"]
  }

  egress {
    from_port   = 0
    to_port     = 0
    protocol    = "-1"
    cidr_blocks = ["0.0.0.0/0"]
  }
}

# 9. The Jenkins EC2 Instance
resource "aws_instance" "jenkins_server" {
  ami                    = data.aws_ami.amazon_linux_2023.id
  instance_type          = "t2.micro" # Free tier eligible

  # Automatically attaching the Instance Profile we created earlier!
  iam_instance_profile   = aws_iam_instance_profile.jenkins_profile.name 
  vpc_security_group_ids = [aws_security_group.jenkins_sg.id]

  # Install Java and Jenkins automatically on boot
  user_data = <<-EOF
              #!/bin/bash
              sudo dnf update -y
              sudo wget -O /etc/yum.repos.d/jenkins.repo https://pkg.jenkins.io/redhat-stable/jenkins.repo
              sudo rpm --import https://pkg.jenkins.io/redhat-stable/jenkins.io-2023.key
              sudo dnf upgrade -y
              sudo dnf install java-17-amazon-corretto -y
              sudo dnf install jenkins -y
              sudo systemctl enable jenkins
              sudo systemctl start jenkins
              EOF

  tags = {
    Name = "Jenkins-Server-Scenario-03"
  }
}

# 10. Output the Jenkins URL so you can easily access it
output "jenkins_url" {
  value       = "http://${aws_instance.jenkins_server.public_dns}:8080"
  description = "Access the Jenkins UI here (Wait 2-3 minutes for installation)"

The "command not found" error occurs because Amazon Linux 2023 (AL2023) has moved from iptables to nftables by default. However, for your DevOps scenario, you can easily install the iptables compatibility layer to make your commands work.

Here is the Perfect Solution to get your Jenkins instance accessible on port 80.

Step 1: Install the missing tool
Run this command in your EC2 terminal to install the iptables package:

Bash
sudo dnf install iptables-services -y
Step 2: Enable IP Forwarding
For the redirection to work at the kernel level, you must ensure IPv4 forwarding is enabled:

Bash
echo 1 | sudo tee /proc/sys/net/ipv4/ip_forward
Step 3: Redirect Port 80 to 8080
Now that the command is installed, run the redirection. This tells the Linux kernel: "If anyone hits port 80, send them to 8080 internally."

Bash
sudo iptables -t nat -A PREROUTING -p tcp --dport 80 -j REDIRECT --to-port 8080
Step 4: Save the Rules (Persistence)
By default, iptables rules disappear after a reboot. Save them so they stay active:

Bash
sudo service iptables save
sudo systemctl enable iptables
sudo systemctl start iptables
Step 5: The "AWS Barrier" (Critical)
Your server is now ready to receive traffic on port 80, but AWS will still block it unless the Security Group is updated.

Go to the AWS Console > EC2 > Instances.

Select your instance and click the Security tab.

Click the Security Group ID.

Click Edit inbound rules and add this rule:

Type: HTTP

Port Range: 80

Source: 0.0.0.0/0

Click Save rules.
